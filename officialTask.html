Final Project: Client Insight Pack

Your final task is to deliver a client-ready insight pack based on the brief you selected. This page is intentionally generic so it applies to every brief.

You are working as a Junior Analyst Intern. The goal is not to show everything you can do, but to deliver a clear, defensible, stakeholder-friendly output that answers the brief.

What you must submit

Submit three things (unless your brief states otherwise):

Component What it is What we look for
1) Dashboard An interactive dashboard that lets a stakeholder explore the data and answer the brief’s key questions. It must include sensible filters and clear labels. Works end-to-end, readable, correct, not cluttered, supports decisions.
2) Short findings note A 1–2 page summary written for non-technical readers: what you found, what it means, and what you recommend next. Clear narrative, cautious claims, actionable recommendations.
3) Robust codebase A fully commented codebase that is logically organised and backed up using Git. Decisions are clear, structure is clean.
4) Technical appendix A short technical record of how you built the work: data sources, cleaning steps, key definitions, limitations, and how to reproduce the analysis. Reproducible, transparent assumptions, correct joins/definitions.

Minimum requirements

  • Your dashboard runs without errors and shows the correct data.
  • You include at least 3 core visuals that directly relate to the brief’s questions.
  • You include at least one KPI / headline metric (clearly defined).
  • You explain what you can and cannot conclude from the data.
  • You document cleaning and assumptions in a way someone else can follow.
  • Your file structure is organised and your work can be reproduced.

How to structure your deliverables

Dashboard (suggested sections)

  • Overview: KPIs + one “big picture” chart.
  • Drivers: breakdowns (by group, time, location, category).
  • Deep dive: filters + drill-down.
  • Notes: definitions + limitations.

Visual standards

  • Every chart has a clear title and labelled axes.
  • Units are obvious (currency, %, counts, minutes, etc.).
  • Legend labels are human-readable (no raw variable names).
  • No chart is there “just because”. Each one answers a question.

Findings note (1–2 pages)

  • Context: what the brief asked.
  • Approach: what data you used and how you analysed it.
  • Key findings: 3–6 bullet points, each backed by evidence.
  • Recommendations: 2–4 actions or next steps.
  • Risks/limits: what might be misleading or missing.

Technical appendix (short)

  • Data sources and definitions (data dictionary highlights).
  • Cleaning decisions (missingness, duplicates, joins, outliers).
  • Assumptions and limitations.
  • How to run your code or app (repro steps).

Suggested file structure

Use a structure like this so your submission is easy to mark and easy to reproduce:

project/
├─ data_raw/
├─ data_clean/
├─ scripts/
├─ app/
├─ outputs/
├─ report/
└─ README.md

Marking criteria (high level)

Your grade is based on both technical correctness and client readiness:

Area What “good” looks like
Analytical judgement Answers the brief, picks appropriate visuals, avoids overclaiming, interprets responsibly.
Data integrity Cleaning is sensible and documented, definitions are consistent, joins make sense.
Reproducibility Clear structure, scripts/app run end-to-end, assumptions recorded, README is usable.
Dashboard quality Readable, well-labelled, interactive where it matters, no broken filters, stakeholder-friendly.
Communication Findings note is clear, concise, and actionable; limitations are stated plainly.

Common pitfalls to avoid

  • Dumping many charts with no narrative.
  • Using raw variable names instead of plain language labels.
  • Making causal claims from observational data.
  • Not defining metrics (what exactly is “delay”, “demand”, “risk”, “usage”).
  • Cleaning steps that are not recorded anywhere.
  • Filters that change the numbers but the text still claims the original story.

Optional stretch goals

  • Add a “So what?” panel that updates with the filters.
  • Add a download of filtered data and/or a chart export.
  • Add a small sensitivity check (e.g. excluding outliers, alternate definition).
  • Add a short “next data we wish we had” section.

Submission checklist

  • Dashboard runs and loads cleanly.
  • Findings note included (PDF/HTML/Doc, as specified).
  • Technical appendix included.
  • README explains how to run the project.
  • All files named clearly and organised.
  • No confidential or personal data included beyond what the brief provides.
